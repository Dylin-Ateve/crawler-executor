# 产品定位：crawler-executor

**生效日期**：2026-04-29
**文档层级**：北极星层 / 产品定位
**更新节奏**：少变；仅当系统群职责边界或产品定位发生变化时更新。

## 1. 系统位置

crawler-executor 是企业级内容生产系统群中的 **第二类：抓取执行系统（Fetching & Rendering）** 的实现仓库。

本系统的核心职责是：把上游下发的 URL 抓取指令转化为原始响应事实，包括 HTTP 响应体、Headers、抓取时延、失败原因、出口 IP、对象存储引用与抓取尝试事件。

本系统从 Scrapy 技术栈起步，独立规划和演进；不继承 Heritrix 抓取链路，也不复用 Heritrix 时期的自动 follow、抓取决策、解析派发等职责渗透行为。

## 2. 服务对象

- **上游第六类（调度与决策）**：本系统消费其下发的抓取指令、抓取参数、队列任务与控制策略。
- **下游第三类（内容加工）**：第三类订阅抓取结果事件，并按 `storage_key` 自取原始 HTML 字节进入加工流水线。
- **下游第五类（画像与状态）**：第五类订阅抓取事实事件，投影抓取历史、URL 状态、Host/Site 画像和审计事实。
- **控制平面运营人员**：通过控制平面下发策略配置、紧急停止、HostGroup 干预与运行参数调整。

## 3. 核心目标

1. **执行 URL 抓取指令**：按上游传入的 URL 与参数完成抓取，不在本系统内自行决定是否抓、何时抓、抓多深。
2. **可靠落盘原始字节**：HTML 原始内容写入对象存储，并对外提供可寻址、可校验、可重放的快照引用。
3. **发布完整抓取事实**：每一次抓取尝试都形成单一 `crawl_attempt` 事件，成功、失败、跳过均有明确结果语义。
4. **支持多出口 IP 执行**：充分利用节点本地辅助 IP 与独立 EIP，降低单出口被封风险。
5. **保持执行层可运维**：提供抓取速率、错误分布、出口 IP 健康、对象存储与事件总线发布状态等执行级指标。

## 4. 永久非目标

这些职责基于系统群边界，永远不属于 crawler-executor：

- 不选择 URL；候选 URL 由第一类产出，是否抓取由第六类裁定。
- 不决定抓取优先级、抓取频率和重抓窗口；这些由第六类负责。
- 不做 HTML 结构化抽取；标题、正文、作者、发布时间等归第三类。
- 不评估内容质量；质量分、分类标签、唯一性判断归第四类。
- 不持有事实层；URL 状态机、域名画像、内容指纹库、抓取历史累积态归第五类。
- 不自动 follow 链接；页面中发现的链接不在本系统内 enqueue。
- 不自管 Redis URL 队列写入；Redis 队列写入侧归第六类，本系统只读消费。
- 不派发解析任务；第三类直接订阅 `crawl_attempt` 事件。

## 5. 阶段性非目标

这些能力当前阶段不做，但未来可在第二类内部演进补齐，不应新建平行执行系统：

- JS 渲染、浏览器执行、现代前端页面的渲染型抓取。
- TLS 指纹、浏览器指纹、动态 UA 与更复杂反爬对抗。
- 控制平面策略运行时覆盖的完整闭环。
- K8s DaemonSet + hostNetwork 的生产化部署。
- 多节点 scrapy-redis 只读消费形态。
- 本地出站事件缓冲、重放和更完整的发布补偿机制。

## 6. 价值主张

crawler-executor 的设计取向是：**哑、快、可横向扩展、字节流可靠落盘**。

- **哑**：执行层不嵌入选择、优先级、解析、质量评价和事实层决策。
- **快**：基于 Scrapy 异步模型和多出口 IP 执行，优先保证吞吐与稳定抓取。
- **可横向扩展**：节点按 DaemonSet / hostNetwork 形态扩展，执行体本身不绑定复杂状态。
- **字节流可靠落盘**：HTML 原始字节进入对象存储，事件只携带引用和校验事实。

## 7. Gate 语义

新 spec 创建时必须检查本文件：

- 是否仍然服务于第二类抓取执行系统的定位。
- 是否把永久非目标误写进本系统范围。
- 是否把阶段性非目标误建成第二套平行执行系统。
- 是否保持“执行层尽量哑”的产品约束。
